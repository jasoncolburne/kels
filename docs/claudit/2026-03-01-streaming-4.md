## High Priority

### 1. `trace_chain_backward_to_serial` and `trace_establishment_backward` have no iteration bound

**Files:** `services/kels/src/handlers.rs:609-633`, `services/kels/src/handlers.rs:678-704`

All five forward-scanning helpers in `handlers.rs` (`recovery_revealed_in_divergent_events`, `scan_divergent_events`, `verify_chain_before_serial`, `non_contest_recovery_revealed_since`, and the divergent event collection loop in `handle_overlap_submission`) were given `max_verification_pages()` guards in round 3. However, the two **backward-walking** helpers were not:

- `trace_chain_backward_to_serial` follows `event.event.previous` pointers from `start_said` back to `stop_serial` with no iteration cap.
- `trace_establishment_backward` follows `event.event.previous` pointers until finding an establishment event with no iteration cap.

A corrupted or malicious database that introduces a cycle in `previous` pointers would cause an infinite loop. This loop holds the PostgreSQL advisory lock, blocking all other submissions for the same prefix indefinitely.

**Why it matters:** The forward-scanning functions were bounded specifically because the DB cannot be trusted (round 3, items 3-4). The same reasoning applies to backward walks. The divergent region is typically small (1-10 events), so a reasonable cap (e.g., `max_verification_pages() * MAX_EVENTS_PER_KEL_QUERY`) would provide safety without affecting normal operation.

## Medium Priority

### 2. `EventNotFound` semantic mismatch for HSM key operations

**Files:** `lib/kels/src/hardware/provider.rs:240`, `services/identity/src/hsm.rs:342,616,624,643`

The rename from `KeyNotFound` to `EventNotFound` was semantically about KEL events (the event SAID was not found), but HSM/hardware code still uses the variant for cryptographic key handle lookups:

- `hardware/provider.rs:240` — `EventNotFound("Public key not found")`
- `identity/hsm.rs:342` — `EventNotFound("Current key not found")`
- `identity/hsm.rs:616,624,643` — `EventNotFound(handle.as_str())` in mock HSM

If downstream error-handling code matches on `EventNotFound` to decide behavior (e.g., falling back to a full fetch in sync code), an HSM key lookup failure could be misinterpreted as a missing event, triggering an incorrect retry instead of a hard error.

**Why it matters:** These errors cross domain boundaries. `HsmError::KeyNotFound` was intentionally preserved for the HSM service itself; the identity-service and hardware-provider HSM code should use a similar dedicated variant rather than the repurposed `EventNotFound`.

### 3. `StoreKelSource::fetch_page` loads entire KEL into memory for `since` resolution

**File:** `lib/kels/src/types/verifier.rs:~675-695`

When `since` is `Some`, `StoreKelSource` calls `self.0.load(prefix, u64::MAX, 0)` — loading the entire KEL into memory just to find the offset of the `since` SAID by linear scan. The code has a warning doc comment ("only suitable for dev-tools/CLI"), but there is no compile-time or runtime guard preventing production use.

**Why it matters:** An adversary who creates a very long KEL could trigger OOM on any node that uses this code path. Consider adding a hard limit to the load or having the backing store support `since`-based queries natively (as the HTTP source does).

### 4. `sync_own_kel` slices `all_events[raft_event_count..]` — index/count mismatch risk

**File:** `services/kels-registry/src/federation/sync.rs:86-100`

The code fetches all identity events via pagination, gets `raft_event_count` from `ctx.event_count()`, then slices `all_events[raft_event_count..]`. This assumes that `event_count()` returns the exact number of events already in Raft and that the flattened event vector is indexed identically to Raft's view. If the identity service returns events in a different order or if `event_count()` doesn't match the index into the flattened list (e.g., if divergent events affect counting), this could panic on out-of-bounds access.

**Why it matters:** A panic would crash the sync loop. The guard `identity_count > raft_event_count` prevents the slice when counts match, but if `event_count()` ever exceeds `all_events.len()` (e.g., Raft has more events than identity returned on this fetch), the guard fails silently (no events submitted) rather than detecting the anomaly.

### 5. `cmd_scheduled_rotate` only fetches first page of KEL events

**File:** `services/identity/src/bin/admin.rs:300-312`

After rotation, the admin tool fetches `GET /api/identity/kel` and deserializes a single `SignedKeyEventPage`. It submits only `page.events` to KELS without checking `page.has_more`. If the identity KEL exceeds 512 events, only the first page is submitted, which would fail KELS verification (missing later events).

**Why it matters:** Identity KELs are typically small, but the tool should either paginate or warn if `has_more` is true. The background sync loop handles this correctly with pagination, so the impact is limited to the admin CLI path.

### 6. `i64::MAX as u64` used as "load all" sentinel in production code paths

**Files:** `lib/kels-ffi/src/lib.rs`, `lib/kels/src/builder.rs`

The refactored `KelStore::load()` uses `limit: u64`, but several production code paths (FFI, builder) pass `i64::MAX as u64` as a "load everything" sentinel, bypassing pagination entirely. For large KELs this loads the entire chain into memory, defeating the purpose of the paginated API.

**Why it matters:** Consider adding a named constant like `LOAD_ALL` to make the intent explicit and searchable, and ensure these paths are only used where unbounded loading is acceptable (CLI, tests). The FFI path in particular serves external consumers who may have large KELs.

## Low Priority

### 7. `build_page_bytes` manually constructs JSON via byte concatenation

**File:** `services/kels/src/handlers.rs`

The `build_page_bytes` function constructs JSON by concatenating `b"{\"events\":"` + cached bytes + `b",\"hasMore\":false}"`. If `SignedKeyEventPage` ever gains new fields, this manually-constructed JSON would silently omit them.

**Why it matters:** Technical debt. The optimization avoids a deserialization/re-serialization round-trip for cached data, which is valid, but creates a coupling between the struct definition and the manual byte construction.

### 8. `ensure_own_kel_synced` uses effective SAID (possibly composite) as `since` cursor

**File:** `services/kels-registry/src/handlers.rs:398-402`

The function gets `raft_effective_said` from `ctx.effective_tail_said()` and uses it as the `since` parameter for `get_key_events()`. If the identity KEL has diverged, the effective SAID would be a composite hash that doesn't correspond to any real event SAID, causing the identity service to return a "not found" error. The code handles this gracefully (`Err(_) => break`), degrading to a no-op.

**Why it matters:** In the divergent case, the KEL sync would silently stop working. Since member KELs are not supposed to diverge, this is a theoretical concern.

### 9. Duplicated `get_signed_history_since` and `compute_prefix_effective_said` implementations

**Files:** `services/identity/src/repository.rs`, `services/kels-registry/src/raft_store.rs`, `services/kels/src/repository.rs`

The same pagination logic (scalar subquery, limit+2 fetch, filter-and-truncate pattern, signature assembly) is duplicated across three repository implementations. Similarly, all three implement the `KelServer` trait with the same delegation pattern.

**Why it matters:** A bug fix in one implementation could be missed in the others. This is a strong candidate for extraction into the `SignedEvents` derive macro or a shared module.

### 10. `trace_chain_backward_to_serial` performs one DB query per event

**File:** `services/kels/src/handlers.rs:617-629`

The backward chain trace walks one event at a time, issuing one `get_event_by_said` query per step. For long owner branches in the divergent region, this is O(N) queries while holding the advisory lock.

**Why it matters:** Divergent regions are typically small (1-10 events), so impact is negligible in practice. A batch query could improve this but adds complexity.

### 11. Recovery hash carried forward without explicit non-reveal check

**File:** `lib/kels/src/types/verifier.rs:~449-477`

When processing an establishment event that does NOT reveal the recovery key, `verify_chain_event` updates `pending_recovery_hash` with whatever `event.recovery_hash` contains (line 477). There is no check that a non-recovery-revealing establishment event must NOT change the recovery hash. The old commitment is silently replaced.

**Why it matters:** This is correct if `validate_structure()` enforces that non-recovery-revealing events cannot change the recovery hash. If it doesn't, a normal `rot` could silently redirect recovery authority without revealing the old key.

### 12. Dead-code guard in `verify_generation` branch-keeping logic

**File:** `lib/kels/src/types/verifier.rs:~361-368`

The `if new_branches.is_empty()` check in the un-extended branch loop would only trigger if zero events matched any branch, which would have already returned an error from the matching loop above. The guard is defensive but is effectively dead code.

**Why it matters:** Minor readability concern. The logic is correct; the guard just can't be reached.

### 13. `CollectSink` uses `tokio::sync::RwLock` for single-writer pattern

**File:** `lib/kels/src/types/verifier.rs:~858-885`

`CollectSink` wraps `Vec<SignedKeyEvent>` in `tokio::sync::RwLock` but is only ever used by a single writer (`transfer_key_events`). A `Mutex` would be more appropriate and cheaper.

**Why it matters:** Minor runtime overhead from uncontended RwLock bookkeeping. Low impact.

### 14. Full `SignedKeyEvent` clone on every verification step

**File:** `lib/kels/src/types/verifier.rs:~423-490`

Each call to `verify_chain_event` clones the full `SignedKeyEvent` (including serialized data and signatures) into `BranchState`. For very long KELs, this adds memory pressure. The `establishment_tip` is also cloned on every non-establishment event.

**Why it matters:** For typical KELs this is negligible. For million-event KELs, consider using `Arc<SignedKeyEvent>` or keeping only the fields needed for branch tracking (SAID, public key, hashes) rather than the full event.

## Positive Observations

- **Verification invariant enforcement is thorough:** `Verification` has `#[crate_new]` making `new()` `pub(crate)` — the only production path to obtain one is `KelVerifier::into_verification()`. The type system prevents consuming unverified KEL data.
- **Fail-secure behavior is consistent:** `truncate_incomplete_generation`, `max_pages` limit enforcement, and recovery/contest logic all default to the most restrictive outcome when state is ambiguous.
- **Comprehensive test replacement:** The deleted `Kel` type (3117 lines of tests) is replaced by `KelVerifier` with 3602 lines of tests covering paginated verification, divergence at page boundaries, anchor checking across pages, resume/incremental verification, and `truncate_incomplete_generation` edge cases.
- **Clean streaming architecture:** `PagedKelSource`/`PagedKelSink`, `transfer_key_events`, `completed_verification`, and `KelServer` provide clear abstractions for pagination throughout the system.
- **Advisory lock discipline is well-maintained:** The submit handler acquires an advisory lock before verification and holds it through insertion, preventing TOCTOU races.
- **Sensitive information redacted from logs:** Redis URLs are no longer logged, preventing credential leakage.
- **Action-oriented error naming:** `RecoveryProtected` → `ContestRequired`, `Rejected` → `RecoverRequired` — the API now tells clients exactly what action to take.
- **Deterministic ordering maintained:** All paginated queries use `ORDER BY serial ASC, CASE kind ... END ASC, said ASC` consistently.
- **`EventKind` enum methods eliminate hardcoded strings:** `sort_priority_mapping()`, `establishment_kinds()`, `as_str()` prevent string-literal errors in SQL.
