## Medium Priority

### ~~2. `EventNotFound` semantic mismatch for HSM key operations~~ RESOLVED

Added `KelsError::HsmKeyNotFound(String)` variant to `lib/kels/src/error.rs`. Updated all 5 HSM-related usages (`hardware/provider.rs:240`, `identity/hsm.rs:342,616,624,643`) from `EventNotFound` to `HsmKeyNotFound`. Added FFI mapping for the new variant (`HsmKeyNotFound` → `KelsStatus::Error`).

### ~~3. `StoreKelSource::fetch_page` loads entire KEL into memory for `since` resolution~~ NOT A PRODUCTION CONCERN

Verified that `StoreKelSource` is only used in `#[cfg(feature = "dev-tools")]` CLI code paths (`cmd_dev_dump_kel`, `cmd_adversary_inject`). Not used in any server-side or production code. The doc comment warning is accurate and sufficient.

### ~~4. `sync_own_kel` loads full KEL into memory and slices by event count index~~ RESOLVED

The original code had two problems: (1) it accumulated the entire identity KEL into memory (`all_events`) then sliced by `raft_event_count` as a vector index, and (2) the event count doesn't correspond to a flat vector index when divergence occurs (multiple events at the same serial). Rewrote `sync_own_kel` to use `effective_tail_said()` as a `since` cursor for delta fetching (same pattern as `ensure_own_kel_synced`). Each page is submitted to Raft individually — no full-KEL-in-memory accumulation, no index-based slicing, and divergence is handled correctly by the SAID-based cursor.

### ~~5. `cmd_scheduled_rotate` only fetches first page of KEL events~~ RESOLVED

Replaced single-page fetch with a pagination loop using `since` cursor and `limit` query parameters. All pages are accumulated into `all_events` before submitting to KELS. Uses `kels::MAX_EVENTS_PER_KEL_RESPONSE` as the page size.

### ~~6. `i64::MAX as u64` used as "load all" sentinel in client-side code paths~~ RESOLVED

Added `pub const LOAD_ALL: u64 = i64::MAX as u64` to `lib/kels/src/lib.rs`. Replaced all caller-side `i64::MAX as u64` usages (builder, FFI, store tests, `StoreKelSource`) with `LOAD_ALL` / `crate::LOAD_ALL` / `kels::LOAD_ALL`. Repository-internal PostgreSQL LIMIT clamping (`limit.min(i64::MAX as u64 - N)`) left as-is — those are defensive bounds, not "load all" semantics.

## Low Priority

### 7. `build_page_bytes` manually constructs JSON via byte concatenation

**File:** `services/kels/src/handlers.rs`

The `build_page_bytes` function constructs JSON by concatenating `b"{\"events\":"` + cached bytes + `b",\"hasMore\":false}"`. If `SignedKeyEventPage` ever gains new fields, this manually-constructed JSON would silently omit them.

**Why it matters:** Technical debt. The optimization avoids a deserialization/re-serialization round-trip for cached data, which is valid, but creates a coupling between the struct definition and the manual byte construction.

### ~~8. `ensure_own_kel_synced` uses effective SAID (possibly composite) as `since` cursor~~ RESOLVED

Resolved by decoupling member KELs from Raft. `ensure_own_kel_synced` now uses `forward_key_events` with `HttpKelSource` (identity) → `RepositoryKelStore` (local MemberKelRepository), which handles pagination internally without using effective SAID as a cursor. See `docs/design/recovery-workflow.md`.

### 9. Duplicated `get_signed_history_since` and `compute_prefix_effective_said` implementations

**Files:** `services/identity/src/repository.rs`, `services/kels-registry/src/raft_store.rs`, `services/kels/src/repository.rs`

The same pagination logic (scalar subquery, limit+2 fetch, filter-and-truncate pattern, signature assembly) is duplicated across three repository implementations. Similarly, all three implement the `KelServer` trait with the same delegation pattern.

**Why it matters:** A bug fix in one implementation could be missed in the others. This is a strong candidate for extraction into the `SignedEvents` derive macro or a shared module.

### 10. `trace_chain_backward_to_serial` performs one DB query per event

**File:** `services/kels/src/handlers.rs:617-629`

The backward chain trace walks one event at a time, issuing one `get_event_by_said` query per step. For long owner branches in the divergent region, this is O(N) queries while holding the advisory lock.

**Why it matters:** Divergent regions are typically small (1-10 events), so impact is negligible in practice. A batch query could improve this but adds complexity.

### 11. Recovery hash carried forward without explicit non-reveal check

**File:** `lib/kels/src/types/verifier.rs:~449-477`

When processing an establishment event that does NOT reveal the recovery key, `verify_chain_event` updates `pending_recovery_hash` with whatever `event.recovery_hash` contains (line 477). There is no check that a non-recovery-revealing establishment event must NOT change the recovery hash. The old commitment is silently replaced.

**Why it matters:** This is correct if `validate_structure()` enforces that non-recovery-revealing events cannot change the recovery hash. If it doesn't, a normal `rot` could silently redirect recovery authority without revealing the old key.

### 12. Dead-code guard in `verify_generation` branch-keeping logic

**File:** `lib/kels/src/types/verifier.rs:~361-368`

The `if new_branches.is_empty()` check in the un-extended branch loop would only trigger if zero events matched any branch, which would have already returned an error from the matching loop above. The guard is defensive but is effectively dead code.

**Why it matters:** Minor readability concern. The logic is correct; the guard just can't be reached.

### 13. `CollectSink` uses `tokio::sync::RwLock` for single-writer pattern

**File:** `lib/kels/src/types/verifier.rs:~858-885`

`CollectSink` wraps `Vec<SignedKeyEvent>` in `tokio::sync::RwLock` but is only ever used by a single writer (`transfer_key_events`). A `Mutex` would be more appropriate and cheaper.

**Why it matters:** Minor runtime overhead from uncontended RwLock bookkeeping. Low impact.

### 14. Full `SignedKeyEvent` clone on every verification step

**File:** `lib/kels/src/types/verifier.rs:~423-490`

Each call to `verify_chain_event` clones the full `SignedKeyEvent` (including serialized data and signatures) into `BranchState`. For very long KELs, this adds memory pressure. The `establishment_tip` is also cloned on every non-establishment event.

**Why it matters:** For typical KELs this is negligible. For million-event KELs, consider using `Arc<SignedKeyEvent>` or keeping only the fields needed for branch tracking (SAID, public key, hashes) rather than the full event.

## Positive Observations

- **Verification invariant enforcement is thorough:** `Verification` has `#[crate_new]` making `new()` `pub(crate)` — the only production path to obtain one is `KelVerifier::into_verification()`. The type system prevents consuming unverified KEL data.
- **Fail-secure behavior is consistent:** `truncate_incomplete_generation`, `max_pages` limit enforcement, and recovery/contest logic all default to the most restrictive outcome when state is ambiguous.
- **Comprehensive test replacement:** The deleted `Kel` type (3117 lines of tests) is replaced by `KelVerifier` with 3602 lines of tests covering paginated verification, divergence at page boundaries, anchor checking across pages, resume/incremental verification, and `truncate_incomplete_generation` edge cases.
- **Clean streaming architecture:** `PagedKelSource`/`PagedKelSink`, `transfer_key_events`, `completed_verification`, and `KelServer` provide clear abstractions for pagination throughout the system.
- **Advisory lock discipline is well-maintained:** The submit handler acquires an advisory lock before verification and holds it through insertion, preventing TOCTOU races.
- **Sensitive information redacted from logs:** Redis URLs are no longer logged, preventing credential leakage.
- **Action-oriented error naming:** `RecoveryProtected` → `ContestRequired`, `Rejected` → `RecoverRequired` — the API now tells clients exactly what action to take.
- **Deterministic ordering maintained:** All paginated queries use `ORDER BY serial ASC, CASE kind ... END ASC, said ASC` consistently.
- **`EventKind` enum methods eliminate hardcoded strings:** `sort_priority_mapping()`, `establishment_kinds()`, `as_str()` prevent string-literal errors in SQL.
