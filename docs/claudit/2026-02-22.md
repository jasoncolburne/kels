## Issues

### 1. `TRUSTED_REGISTRY_MEMBERS` format missing `active` field

**Severity**: High — will cause deployment failures
**Affected files**: `docs/federation.md` (line 71), `docs/deployment.md` (line 106)

Multiple docs describe `TRUSTED_REGISTRY_MEMBERS` as a JSON array of `{id, prefix}` objects. The code requires an `active` boolean field:

```rust
// services/kels-registry/src/federation/config.rs:17-20
struct TrustedMember {
    id: FederationNodeId,
    prefix: String,
    active: bool,  // <-- required, not documented
}
```

A test at `config.rs:606` explicitly verifies that missing `active` causes deserialization failure.

**Fix**: Update all examples and descriptions to `{id, prefix, active}`, e.g.:
```
TRUSTED_REGISTRY_MEMBERS='[{"id":1,"prefix":"ERegistryAcme...","active":true}]'
```

### 2. `owner_tail` tracking references non-existent API

**Severity**: Medium — documents feature that doesn't exist as described
**Affected files**: `docs/divergence-detection.md` (line 40), `docs/protocol-attack-surface.md` (line 143)

Both files reference `save_owner_tail()` / `load_owner_tail()` methods on `KelStore`. These methods do not exist on the `KelStore` trait (`lib/kels/src/store/mod.rs`). The trait only provides `load`, `save`, `delete`, and `cache`.

What does exist:
- `FileKelStore::owner_tail_path()` — a path helper, but never called by save/load
- `KeyEventBuilder::get_owner_tail()` (`lib/kels/src/builder.rs:472`) — returns the last event in the builder's local KEL, not a persisted SAID

The underlying concept (tracking the owner's branch tip for recovery chaining) works through the builder's in-memory KEL state, not through a dedicated persisted `owner_tail` store.

**Fix**: Update the docs to describe the actual mechanism (builder tracks local KEL tail in memory) or implement the persisted API if it's intended.
