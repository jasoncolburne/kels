## High Priority

### ~~1. Anchor checking no longer excludes divergent events â€” security regression~~ RESOLVED

Added `diverged_at_serial` guard in `verify_generation`: anchors at or after the divergence serial are not recorded in `anchored_saids`. Pre-divergence anchors remain trusted. Regression tests: `test_anchors_on_divergent_branches_excluded`, `test_anchors_after_divergence_excluded_multi_serial`.

### ~~2. `completed_verification` does not fail secure on `max_pages` exhaustion~~ RESOLVED

Added `exhausted` flag tracking. When the loop exits due to `max_pages` exhaustion (not source depletion), returns `Err(KelsError::InvalidKel(...))` instead of a partial `Verification`. Regression tests: `test_max_pages_limit_fails_secure`, `test_max_pages_exact_boundary_succeeds`, `test_max_pages_one_over_boundary_fails`.

### ~~3. Gossip sync only fetches the first page of delta events~~ RESOLVED

Added `KelsClient::fetch_all_key_events()` which paginates through all pages with a `max_pages` guard (fails if exceeded). Replaced all 5 `fetch_key_events` call sites in gossip `handle_announcement` with `fetch_all_key_events`.

### ~~4. Unbounded pagination loops in divergence scan helpers~~ RESOLVED

Added `max_verification_pages()` guard to all 5 scan loops in `handlers.rs`: `recovery_revealed_in_divergent_events`, `scan_divergent_events`, `verify_chain_before_serial`, `non_contest_recovery_revealed_since`, and the divergent event collection loop in `handle_overlap_submission`. All fail secure with an error on exhaustion.

## Medium Priority

### 5. Every submission pays full `completed_verification` cost

**File:** `services/kels/src/handlers.rs` (submit handler)

Every event submission under advisory lock now verifies the entire KEL via paginated `completed_verification`. For a 100K-event KEL, this means ~200 pages of 512 events each, with full cryptographic verification. The fast path for normal appends uses `KelVerifier::resume()`, but the initial `completed_verification()` is still called unconditionally.

**Recommendation:** Consider caching `Verification` tokens in Redis (keyed by prefix + effective SAID) so subsequent submissions can resume from a cached token.

### 6. `apply_submit_key_events` DB persist errors logged at `debug` level

**File:** `services/kels-registry/src/federation/state_machine.rs`

DB persistence errors are silently swallowed at `debug` level. If the DB write genuinely fails (not a duplicate key), the Raft state will have updated `member_contexts` but the DB will be missing events, permanently blocking that prefix.

**Recommendation:** Log at `warn` level. Consider distinguishing duplicate-key errors (expected on Raft log replay) from other storage errors.

### 7. Cache hit in `get_kel` deserializes and re-serializes unnecessarily

**File:** `services/kels/src/handlers.rs` (`get_kel`)

The single-prefix `get_kel` handler deserializes cached bytes back into `Vec<SignedKeyEvent>`, wraps in `SignedKeyEventPage`, and lets axum re-serialize. The batch handler `get_kels_batch` avoids this with `build_page_bytes`.

**Recommendation:** Use the same `build_page_bytes` approach in `get_kel` for the hot path.

### 8. `raft_event_count` computed from `tip.serial + 1` assumes contiguous zero-based serials

**Files:** `services/kels-registry/src/federation/sync.rs` (`sync_own_kel`), `services/kels-registry/src/handlers.rs` (`admin_vote_proposal`)

The old code used `k.events().len()` which was accurate. `serial + 1` may not equal the actual event count after recovery or if serials have gaps.

**Recommendation:** Add a comment documenting the assumption or compute a more robust count.

### 9. No integration tests for divergence/recovery/contest in refactored handlers

**File:** `services/kels/src/handlers.rs`

The `handle_divergent_submission` and `handle_overlap_submission` are substantially different from the old `Kel::merge`. The old merge tests were removed, but no integration tests cover these new security-critical code paths.

**Recommendation:** Add integration tests exercising divergence, recovery, contest, and overlap submission through the HTTP handler.

## Low Priority

### 10. Import ordering violation in `handlers.rs`

**File:** `services/kels/src/handlers.rs`

The `kels` import block precedes the `std` import block. Per CLAUDE.md, `std` imports should come first.

### 11. `get_effective_said` endpoint returns `serde_json::Value` instead of typed struct

**File:** `services/kels/src/handlers.rs` (`get_effective_said`)

Every other endpoint uses typed response structs. This one uses `serde_json::json!({ "said": said })`.

**Recommendation:** Create a small `EffectiveSaidResponse { said: String }` struct.

### 12. Silent fallback on malformed `KELS_MAX_VERIFICATION_PAGES` env var

**File:** `lib/kels/src/types/verifier.rs` (`max_verification_pages`)

If someone sets `KELS_MAX_VERIFICATION_PAGES=abc`, it silently uses the default.

**Recommendation:** Log a warning when the env var is set but unparseable.

### 13. Variable shadowing of `ctx` in `show_identity_status`

**File:** `services/kels-registry/src/bin/kels-registry-admin.rs`

The function parameter `ctx: &AdminContext` is shadowed by a local `ctx: Option<Verification>`. A reader may confuse the two.

**Recommendation:** Rename the local to `verification` or `identity_ctx`.

### 14. `transfer_key_events` composite SAID cursor fragility

**File:** `lib/kels/src/types/verifier.rs` (`transfer_key_events`)

When transferring a divergent KEL, the `since` cursor between pages is a composite SAID (`hash_tip_saids`). If the KEL state changes between fetches (e.g., recovery), the composite SAID becomes invalid and the server returns `KeyNotFound`.

**Recommendation:** Document this assumption more explicitly or add error handling for the stale-cursor case.

### 15. `StoreKelSource::fetch_page` loads entire KEL into memory for `since`-based pagination

**File:** `lib/kels/src/types/verifier.rs` (`StoreKelSource`)

When `since` is `Some`, loads all events via `u64::MAX` limit then scans linearly. Only used in dev-tools CLI so not a production concern.

**Recommendation:** Add a prominent doc comment warning this is O(N) and unsuitable for production.

## Positive Observations

- **Verification invariant well enforced:** `Verification` has no public constructor â€” only obtainable via `KelVerifier::into_verification()`. Type system prevents consuming unverified KEL data.
- **Clean API design:** `SignedKeyEventPage`, `PagedKelSource`/`PagedKelSink`, `KelServer` trait provide clear abstractions for pagination and streaming.
- **Deterministic ordering maintained:** All paginated queries use `ORDER BY serial ASC, CASE kind ... END ASC, said ASC` consistently.
- **Good test coverage for new paths:** Multi-page verification, anchor checking across pages, divergence at page boundaries, transfer with held-back events, partition-for-submission â€” all well tested.
- **`compute_prefix_effective_said` uses efficient SQL:** `NOT EXISTS` correlated subquery with proper index (`idx_kels_key_events_prefix_previous_idx`) avoids loading all events.
- **Recovery path double-checks:** Belt-and-suspenders approach in `handle_divergent_submission` with both `recovery_revealed_in_divergent_events` and `scan_divergent_events` checking adversary recovery independently.
