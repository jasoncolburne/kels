**File:** `services/identity/src/hsm.rs:146-149`

~~The `generate_keypair` method uses a catch-all `_` arm for the algorithm string that silently generates ML-DSA-65 keys for any unrecognized value, including typos like `"ml-dsa87"` or `"ML-DSA-87"`.~~

### ~~10. `HsmKeyProvider::Clone` uses `blocking_read()` — panics in async context~~ — RESOLVED

**File:** `services/identity/src/hsm.rs:288-306`

~~The `Clone` implementation calls `self.signing_algorithm.blocking_read()` and similar on all `RwLock` fields. Tokio's `blocking_read()` panics if called from within an async runtime thread.~~

**Resolution:** Removed the `Clone` impl from `HsmKeyProvider` and added `pub async fn clone_async(&self)` which uses `.read().await`. ## Positive Observations

- **Complete PKCS#11 3.2 mock implementation.** The `kels-mock-hsm` library implements a functional subset of PKCS#11 with proper state machine management (initialized → session open → logged in), correct handle allocation, object search, and ML-DSA-65/ML-DSA-87 key generation and signing. The persistence layer enables testing across service restarts. The stub macro for unsupported functions is clean.

- **Robust streaming verification.** The `KelVerifier` handles both linear and divergent KELs, with proper per-branch cryptographic state tracking, forward commitment validation (rotation hashes), dual-signature requirements for recovery events, and anchor checking. The invariant enforcement (max 2 events per generation, no post-divergence multi-events) catches DB corruption or protocol violations.

- **Fail-secure KEM algorithm negotiation.** The `requires_kem_1024` flag starts `true` and is only relaxed after verifying federation algorithms. The allowlist refresh loop correctly checks all peers' signing algorithms and upgrades to ML-KEM-1024 if any peer uses ML-DSA-87. The acceptor-side validation rejects weaker KEM offers.

- **Well-designed key provider abstraction.** The `KeyProvider` trait cleanly separates software, hardware (Secure Enclave), and HSM-backed implementations with a consistent two-phase rotation protocol (stage/commit/rollback). The `ProviderConfig` and `KeyStateStore` traits enable provider-agnostic persistence.

- **Peer verification is defense-in-depth.** The gossip `KelsPeerVerifier` performs both authorization (allowlist check with one-shot refresh) and authentication (KEL-based signature verification with one-shot re-fetch on mismatch). The retry-once pattern avoids hammering the registry while still handling legitimate key rotations.

- **FFI layer is thorough and safe.** The C FFI properly handles null pointers, thread-local error reporting, algorithm option parsing, and key state persistence across operations. The `parse_algorithm_option` returns `None` for unrecognized inputs (keeping current algorithm) rather than silently changing behavior, and `parse_algorithm` defaults to ML-DSA-65 (fail secure — post-quantum by default).
