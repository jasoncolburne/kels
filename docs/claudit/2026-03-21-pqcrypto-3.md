## Medium Priority

### ~~1. Mock HSM catch-all `_` arms default to ML-DSA-65 for unrecognized parameter sets~~ — RESOLVED

**Files:** `lib/kels-mock-hsm/src/lib.rs:618-628` (C_GenerateKeyPair), `lib/kels-mock-hsm/src/lib.rs:955-966` (C_Sign)

~~Both `C_GenerateKeyPair` and `C_Sign` use a catch-all `_ =>` arm that silently falls through to ML-DSA-65 when the stored `parameter_set` is not `CKP_ML_DSA_87`. If a caller passes an unrecognized `CKA_PARAMETER_SET` value, key generation silently creates ML-DSA-65 keys and signing silently uses ML-DSA-65. This violates the fail-secure principle at the PKCS#11 layer.~~

**Resolution:** Replaced `_ =>` with explicit `CKP_ML_DSA_65 =>` arms and added catch-all `_ =>` returning `CKR_ARGUMENTS_BAD` (key generation) and `CKR_MECHANISM_INVALID` (signing).

### ~~2. Private key files and state files created with default permissions (potentially world-readable)~~ — RESOLVED

**Files:** `lib/kels/src/crypto.rs:156-160` (FileKeyStateStore::save), `lib/kels/src/crypto.rs:434-462` (SoftwareKeyProvider::save_to_dir)

~~Both `FileKeyStateStore::save()` and `SoftwareKeyProvider::save_to_dir()` use `std::fs::create_dir_all()` and `std::fs::write()` without setting restrictive file permissions. On a typical Unix system with umask 022, directories are created with mode 755 (world-listable) and key files with mode 644 (world-readable).~~

**Resolution:** Added `write_key_file()` and `ensure_private_dir()` helpers that set 0o600 (files) and 0o700 (directories) on Unix via `std::os::unix::fs::PermissionsExt`. Applied to both `FileKeyStateStore::save()` and `SoftwareKeyProvider::save_to_dir()`.

## Positive Observations

- **Thorough resolution of prior findings.** All 12 findings from Rounds 1-2 have been addressed with high-quality fixes. The `blocking_read()` replacement with `clone_async()` (Round 2 #10) was particularly well-designed, relaxing the `Clone` bound on `KeyEventBuilder` only where needed rather than adding a blanket requirement.

- **Correct `len-2` indexing pattern across providers.** Both `SoftwareKeyProvider` and `HardwareKeyProvider` use `keys[len-2]` for the current signing key, which is correct for the key lifecycle: `[current, next]` normally, `[old, new_current, new_next]` after staging. The explicit `"this is correct"` comments show this was already reviewed.

- **Robust algorithm inference in `restore_state`.** After deserializing and validating key counts (Round 2 #11 fix), algorithms are inferred from the loaded keys. The `unwrap_or(self.signing_algorithm)` is actually unreachable (validation already proved keys exist), but provides a safe fallback rather than panicking.

- **Clean handshake protocol design.** The ML-KEM handshake correctly derives direction-separated session keys using blake3 KDF with CESR-encoded prefixes (always valid UTF-8). The signature exchange uses a JSON payload binding both peers' ephemeral KEM keys and the peer's prefix, preventing relay attacks.

- **DashMap-based rate limiting is shard-atomic.** The `entry()` API holds the shard lock while the closure executes, so concurrent requests for the same prefix are properly serialized. The rate limiting window reset and increment are atomic within the DashMap shard.

- **Defense-in-depth verification retry.** The gossip `KelsPeerVerifier` tries verification once, then on failure refreshes both the allowlist (one-shot) and the peer's KEL (via forward), then retries. This handles legitimate key rotations without requiring lockstep deployment across nodes.
