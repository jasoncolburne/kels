## Medium Priority

### 1. Weight accumulation can overflow in both parser validation and evaluator

**File:** `lib/kels-policy/src/parser.rs:210` and `lib/kels-policy/src/evaluator.rs:219`

In `parse_weighted`, the total weight validation uses `pairs.iter().map(|(_, w)| w).sum()` which wraps on overflow in release mode (panics in debug). A crafted policy with extreme weights could cause the validator to accept an unsatisfiable policy (if the wrapped sum is less than `min_weight`) or reject a satisfiable one. Similarly, in `evaluate_node`, `total_weight += weight` wraps in release mode.

Practically unlikely (requires adversarial u64 values), but violates the fail-secure principle — the evaluator should never produce wrong answers even on adversarial input.

**Suggested fix:** Use `saturating_add` for the accumulation in both locations. In the parser, consider capping individual weights to a reasonable maximum (e.g., `u32::MAX`) or using `checked_add` with an error on overflow.

### 2. Stale domain separator documentation

**File:** `docs/design/policy.md:113`

The text says "The domain separator is shared with the legacy revocation hash for backward compatibility." This was resolved in round 1 finding #2 — the domain separator was changed to `b"kels/poison:"` (distinct from the old `b"kels/revocation:"`) and `revocation.rs` was deleted entirely. The doc is now misleading.

**Suggested fix:** Replace with something like "The domain separator `kels/poison:` is purpose-named to distinguish poison hashes from other anchored SAIDs."

## Low Priority

### 3. `endorser_prefixes()` omits poison expression endorsers

**File:** `lib/kels-policy/src/policy.rs:111-116`

`Policy::endorser_prefixes()` only parses the main `expression`, not the `poison` expression. If a caller uses this method to pre-fetch KELs for all prefixes involved in policy evaluation, the poison expression's endorsers would be missing, causing evaluation-time KEL fetches.

**Suggested fix:** Also parse `self.poison` (if present) and merge its prefixes into the result set.

### 4. `PolicyNode` derives `PartialEq` but not `Eq`

**File:** `lib/kels-policy/src/ast.rs:4`

`PolicyNode` contains only `String` and `u64`, both of which implement `Eq`. Deriving `Eq` is free and enables use in `HashSet`, `BTreeSet` `.contains()` patterns, and other contexts that require full equivalence. Currently only `PartialEq` is derived.

**Suggested fix:** Add `Eq` to the derive list: `#[derive(Debug, Clone, PartialEq, Eq)]`.

## Positive Observations

- **All 8 round 1 findings are resolved.** Every finding from the first audit was addressed — including the critical canonical policy matching in edge verification, the domain separator fix, and the nested policy evaluation correction. Clean execution across the full list.

- **Fail-secure on compacted delegate evaluation.** If a compacted `delegate(DELEGATOR)` (empty delegate) is evaluated, `verify_delegation` attempts to verify an empty-prefix KEL, which fails. This returns `false` — the secure default. The library doesn't need to special-case this because the secure failure mode handles it naturally.

- **Cycle detection correctly handles fan-out vs cycles.** The visited-set logic in `evaluate_node` adds the policy SAID before recursion and removes it after, which correctly allows the same policy to appear in sibling branches (`threshold(2, [policy(X), policy(X)])`) while still catching true cycles (`X → Y → X`). This is a subtle distinction handled correctly.

- **`deserialize_immune` normalization prevents SAID divergence.** The custom deserializer that normalizes `Some(false)` to `None` prevents a class of bugs where two behaviorally-identical policies would have different SAIDs. Good defensive design at the serialization boundary.

- **Test coverage is thorough and scenario-driven.** The evaluator tests cover single endorser, threshold, weighted, poisoning (standard, proactive, immune, poison-expression with admin, non-admin, threshold), nested policy, cycle detection, and delegation. The credential integration tests cover 3-level credential chains with edge verification. This is well beyond typical unit test coverage.

- **Clean separation between kels-policy and kels-creds.** The kels-creds library depends on kels-policy via Cargo dependency and re-exports policy types for convenience, but the policy evaluation logic is entirely self-contained. This enables independent testing and future reuse outside the credential context.
