## High Priority

### ~~1. SAD chain verifier skips SAID and signature verification for divergent records~~ — RESOLVED

**File:** `lib/kels/src/types/sad/verification.rs:67-70`

~~When a duplicate version is detected (divergence), the verifier sets `is_divergent = true` and `continue`s, skipping `verify_said()`, `verify_prefix()`, chain linkage checks, and signature verification for that record. While the divergent record is not stored in `self.tip` or used for security decisions, the `is_divergent` flag itself becomes trusted without cryptographic backing — an attacker could inject a garbage record at a duplicate version to set the flag and influence caller behavior (callers use `is_divergent()` to decide sync strategy).~~

**Resolution:** Moved `verify_said()` and signature verification before the divergence `continue`. The `is_divergent` flag is now only set by cryptographically valid records.

## Medium Priority

### ~~2. `Hash` and `PartialEq` contract violation on `SignedKeyEvent`~~ — RESOLVED

**File:** `lib/kels/src/types/kel/event.rs:584-616`

~~`PartialEq` uses order-independent set comparison (collects signatures into `HashSet`, checks containment), but `Hash` iterates `self.signatures.clone()` in order. Two `SignedKeyEvent`s with the same signatures in different order would be `==` but produce different hashes, violating Rust's `Hash`/`Eq` contract.~~

**Resolution:** `Hash` now sorts signatures canonically (`sort_unstable`) before hashing, matching the order-independent `PartialEq`. Also removed the unnecessary `.clone()`.

### ~~3. Hardcoded event kind strings in identity service~~ — RESOLVED

**File:** `services/identity/src/server.rs:399-458, 488`

~~Uses bare string literals `"ror"`, `"rot"`, `"rec"`, `"cnt"`, `"dec"` for event kinds, violating the CLAUDE.md rule: "Never hardcode event kind strings — use `EventKind` enum methods."~~

**Resolution:** Added `EventKind::short_name()` method to `kel/event.rs`. Replaced all hardcoded strings with `EventKind` enum variants (`EventKind::Ror`, `EventKind::Rot`, etc.). Comparisons use enum equality, logging and response use `.short_name()`.

## Positive Observations

- **`auth.rs` extraction is well-designed.** `validate_timestamp` and `SignedRequest<T>` are cleanly separated from peer/voting concerns. The fail-secure divergence check (`if kel_verification.is_divergent() { return Err(Divergent) }`) is correctly placed before any key usage. The test coverage includes boundary conditions for timestamp validation and a proper divergent-KEL rejection test.

- **Types reorganization preserves public API perfectly.** Each subdirectory (`kel/`, `sad/`, `federation/`, `gossip/`) uses `pub use submodule::*` in its `mod.rs`, so all downstream code continues to import from `kels_core::` without path changes. The `#[allow(clippy::too_many_arguments)]` attributes correctly migrated to the new `kel/mod.rs`.

- **`SadAnnouncement` encapsulation is thorough.** Moved to `services/gossip/src/types/sad.rs` with `pub(crate)` visibility, and all gossip service types (`GossipEvent`, `GossipCommand`, `KelAnnouncement`) consistently use `pub(crate)`. Tests verify serialization round-trip including backwards compatibility for the `repair` field.

- **`NodeInfo`/`NodeType` removal is complete.** Grep confirms zero remaining references across all Rust and Swift source files. `NodeStatus` was correctly relocated to `client/kels.rs` where it's actually used, and re-exported through `lib.rs`.

- **`verify_with_contexts` chain verification is sound.** Despite the name suggesting it validates peer records against KEL data, the actual contract is clear: it checks that all provided KEL verifications belong to trusted prefixes, then verifies structural chain integrity (SAID, previous pointers, version sequence). This two-layer approach correctly separates trust decisions from structural validation.

- **Gossip service visibility audit is clean.** All internal types in the gossip service binary crate use `pub(crate)` — `SadAnnouncement`, `KelAnnouncement`, `GossipEvent`, `GossipCommand`, and the channel types. This prevents accidental public API surface in a binary crate.
